# 要件定義書 - 日報システム

**バージョン**: 1.3  
**作成日**: 2026-05-18  
**更新日**: 2026-05-19  
**ステータス**: ドラフト

---

## 1. システム概要

営業担当者が日々の顧客訪問記録・課題・翌日計画を報告し、上長が確認・承認するための日報管理システム。

---

## 2. 利用者・ロール

| ロール / フラグ         | 説明                                                            |
| ----------------------- | --------------------------------------------------------------- |
| 営業（role: SALES）     | 日報を作成・提出・修正する                                      |
| 上長（role: MANAGER）   | 配下営業の日報を閲覧・承認・差し戻し・コメントする（1階層のみ） |
| 管理者（isAdmin: true） | 顧客マスタ・営業マスタのCRUD操作を行う。上長との兼務可          |

> **上長の階層**: 1階層のみ（上長 → 営業）。上長の上長は本システムの対象外。

---

## 3. 機能要件

### 3.0 認証・ログイン

- メールアドレスとパスワードによるログイン画面を提供する
- ログイン成功後、ロールに応じた画面にリダイレクトする
  - 営業（SALES） → 自分の日報一覧
  - 上長（MANAGER） → 配下営業の提出済み日報一覧
- 管理者画面（顧客一覧・営業一覧）は `isAdmin: true` のユーザーのみナビゲーションメニューから遷移できる。ログイン直後のリダイレクト先にはならない
- ログアウト機能を提供する
- 未ログイン状態でのページアクセスはログイン画面にリダイレクトする

---

### 3.1 日報管理（営業）

#### 3.1.1 日報作成

- 営業は自身の日報を1日につき1件作成できる
- 日付は過去日・当日のみ選択可能（未来日は不可）
- 日報には以下の項目を入力する
  - **日付**（必須）
  - **訪問記録**（任意・複数行）: 顧客マスタから顧客を選択し、訪問内容を記述する
    - 1件の日報につき、訪問記録を複数行追加・削除可能（最大10件）
    - 訪問記録が0件のまま提出することも可能（内勤日など）
    - 訪問記録を1行以上入力する場合、顧客・訪問内容は両方必須
  - **Problem**（課題・相談）: 自由記述（任意）
  - **Plan**（明日やること）: 自由記述（任意）

#### 3.1.2 日報提出

- 作成した日報を上長に提出できる
- 提出するとステータスが「提出済」になる
- 差し戻された日報は再提出できる（内容の修正は任意）

#### 3.1.3 日報一覧・詳細閲覧

- 自分の日報一覧を日付・ステータスでフィルタして閲覧できる
- 日報の詳細（訪問記録・Problem・Plan・上長コメント）を閲覧できる

---

### 3.2 承認・差し戻し（上長）

- 上長は配下の営業が提出した日報を一覧で確認できる
- 日報を**承認**または**差し戻し**できる（いずれも確認ダイアログあり）
  - **承認**: ステータスが「完了」に変わる。取り消し不可。それ以降、営業は編集不可
  - **差し戻し**: ステータスが「差し戻し」に変わる。営業が修正・再提出可能
- 差し戻し時は理由をコメント（targetField: GENERAL）として記入できる（任意）
- 承認・差し戻しが可能なのはステータスが「提出済」の日報のみ

---

### 3.3 コメント機能

- 上長はステータスが**提出済**の日報に対してコメントを入力できる
  - コメントは以下の3種類（`targetField`）がある
    - **PROBLEM**: Problem（課題・相談）へのコメント
    - **PLAN**: Plan（明日やること）へのコメント
    - **GENERAL**: 全般コメント（差し戻し理由などを含む）
- コメントは日報詳細画面で営業も閲覧できる
- コメントは追加のみ可能（送信後の編集・削除は不可）
- 下書き・差し戻し・完了ステータスの日報にはコメント不可

---

### 3.4 パスワード変更

- 認証済みの全ユーザーが自分自身のパスワードを変更できる
- 変更時は現在のパスワードを入力して本人確認を行う
- 新しいパスワードは8文字以上かつ英字・数字を両方含む必要がある

---

### 3.5 マスタ管理（管理者）

管理者は専用の管理画面から顧客マスタ・営業マスタの追加・編集・削除ができる。

#### 顧客マスタ管理

| 操作     | 説明                                                                                   |
| -------- | -------------------------------------------------------------------------------------- |
| 一覧表示 | 顧客名で検索・絞り込み可能                                                             |
| 追加     | 顧客名（必須）、住所・備考（任意）を入力して登録                                       |
| 編集     | 各項目を編集して保存                                                                   |
| 削除     | `isDeleted` フラグをtrueにする論理削除。削除後も過去の訪問記録では顧客名を表示し続ける |

**顧客マスタ項目**

| 項目           | 型      | 必須 | 説明                              |
| -------------- | ------- | ---- | --------------------------------- |
| 顧客ID         | UUID    | -    | システム採番                      |
| 顧客名         | String  | ○    | -                                 |
| 住所           | String  | -    | -                                 |
| 備考           | Text    | -    | -                                 |
| デリートフラグ | Boolean | ○    | デフォルトfalse。trueで論理削除済 |

#### 営業マスタ管理

| 操作     | 説明                                                                                       |
| -------- | ------------------------------------------------------------------------------------------ |
| 一覧表示 | 氏名・ロールで絞り込み可能                                                                 |
| 追加     | 必須項目を入力してアカウント作成（初期パスワード自動発行）                                 |
| 編集     | 氏名・上長・ロールを編集可能                                                               |
| 削除     | `isDeleted` フラグをtrueにする論理削除。削除後も過去の日報・コメントでは氏名を表示し続ける |

**営業マスタ項目**

| 項目           | 型      | 必須           | 説明                                                   |
| -------------- | ------- | -------------- | ------------------------------------------------------ |
| 社員ID         | UUID    | -              | システム採番                                           |
| 氏名           | String  | ○              | -                                                      |
| メールアドレス | String  | ○              | ログインID兼用・一意                                   |
| パスワード     | String  | ○              | ハッシュ保存                                           |
| 上長           | FK      | ○（SALESのみ） | 営業ロールの場合は必須。MANAGERの場合はNULL            |
| ロール         | Enum    | ○              | 営業 / 上長                                            |
| 管理者フラグ   | Boolean | ○              | trueの場合マスタ管理画面にアクセス可（上長との兼務可） |
| デリートフラグ | Boolean | ○              | デフォルトfalse。trueで論理削除済                      |

**営業マスタの制約**

- 管理者は自分自身のアカウントを削除できない
- システム内に MANAGER ロールのユーザーが1名のみの場合、そのユーザーは削除できない
- 上長（MANAGER）が変更・削除された場合、変更後の上長が既存の提出済み日報も承認できる

**パスワードリセット**

- 管理者は営業マスタ編集画面から対象ユーザーのパスワードをリセットし、新しい初期パスワードを発行できる
- 発行された初期パスワードは画面に一度だけ表示する（管理者がユーザーに共有する運用）

## 4. 日報ステータス遷移

```
[下書き] → (提出) → [提出済] → (承認) → [完了]
                           ↓
                        (差し戻し)
                           ↓
                        [差し戻し] → (修正・再提出) → [提出済]
```

| ステータス | 説明     | 営業の操作   | 上長の操作               |
| ---------- | -------- | ------------ | ------------------------ |
| 下書き     | 作成中   | 編集・提出   | -                        |
| 提出済     | 承認待ち | 閲覧のみ     | 承認・差し戻し・コメント |
| 差し戻し   | 要修正   | 編集・再提出 | -                        |
| 完了       | 承認済   | 閲覧のみ     | 閲覧のみ                 |

## 5. データ要件（概要）

### 共通カラム

全テーブルに以下のカラムを持つ。

| カラム       | 型       | 説明                                                      |
| ------------ | -------- | --------------------------------------------------------- |
| lastUpdateId | String   | 更新者のユーザーID、またはシステム処理の場合は `"system"` |
| createdAt    | DateTime | レコード作成日時                                          |
| updatedAt    | DateTime | レコード最終更新日時（Prisma `@updatedAt` で自動管理）    |

### 日報（DailyReport）

| 項目    | 型   | 説明                              |
| ------- | ---- | --------------------------------- |
| id      | UUID | PK                                |
| userId  | FK   | 作成した営業（User.id）           |
| date    | Date | 日報の日付                        |
| problem | Text | 課題・相談                        |
| plan    | Text | 明日やること                      |
| status  | Enum | 下書き / 提出済 / 差し戻し / 完了 |

### 訪問記録（VisitRecord）

| 項目       | 型   | 説明     |
| ---------- | ---- | -------- |
| id         | UUID | PK       |
| reportId   | FK   | 日報ID   |
| customerId | FK   | 顧客ID   |
| content    | Text | 訪問内容 |
| sortOrder  | Int  | 表示順   |

### コメント（Comment）

| 項目        | 型   | 説明                           |
| ----------- | ---- | ------------------------------ |
| id          | UUID | PK                             |
| reportId    | FK   | 日報ID                         |
| authorId    | FK   | コメント者（上長）             |
| targetField | Enum | `PROBLEM` / `PLAN` / `GENERAL` |
| body        | Text | コメント本文                   |

## 6. 技術スタック

| カテゴリ         | 技術                           | 説明                                |
| ---------------- | ------------------------------ | ----------------------------------- |
| Framework        | Next.js 15 (App Router)        | フロント・バック統合                |
| 言語             | TypeScript 5.x                 | 全体                                |
| ORM              | Prisma 6.x                     | DBアクセス                          |
| 認証             | Auth.js (NextAuth v5)          | セッション管理                      |
| DB               | PostgreSQL on Neon             | データ永続化（無料枠 0.5GB）        |
| バリデーション   | Zod                            | APIリクエスト検証                   |
| APIスキーマ      | @asteasolutions/zod-to-openapi | OpenAPI仕様自動生成                 |
| UI               | Tailwind CSS + shadcn/ui       | スタイリング                        |
| 単体・統合テスト | Vitest                         | APIロジック・ユーティリティのテスト |
| E2Eテスト        | Playwright                     | 画面操作・ユーザーフローのテスト    |
| コンテナ         | Docker                         | アプリのコンテナ化                  |
| デプロイ         | Google Cloud Run               | サーバーレスコンテナ実行            |

### APIスキーマ方針

- 各Route HandlerにZodスキーマを定義し、リクエストボディ・クエリパラメータを検証する
- `@asteasolutions/zod-to-openapi` でZodスキーマからOpenAPI 3.0仕様を自動生成する
- `/api/docs` エンドポイントでOpenAPI JSONを配信し、Swagger UIで閲覧可能にする

### テスト方針

| 種別       | ツール     | 対象                                           |
| ---------- | ---------- | ---------------------------------------------- |
| 単体テスト | Vitest     | バリデーションロジック、ユーティリティ関数     |
| 統合テスト | Vitest     | APIエンドポイント（DBありで検証）              |
| E2Eテスト  | Playwright | ログイン・日報提出・承認フローなど主要画面操作 |
